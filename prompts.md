## Step 3: TDD 算法与核心逻辑实现阶段

### Prompt（用户输入）
```
请先阅读项目说明书 `CLAUDE.md` 和图片 `4aec573396557f20b6d6cc5f06ffeda.jpg`，确认你理解了完整的作业要求。本次我们进入【TDD阶段】，请只聚焦于该阶段的任务，不要提前实现后续阶段的内容（尤其是前端 UI 修改、HTTP API 接口和 E2E 端到端测试）。

作业要求我们必须遵循三大范式：SDD（契约/模型驱动）→ DDD（设计驱动）→ TDD（测试驱动）。本阶段是 TDD，必须以"先写测试、再写实现、最后重构"为核心工作流。

### 当前已有基础
- SDD 阶段已产出 `docs/schema.md`，定义了 6 张表结构、Q/X 矩阵导出方式及 API 契约
- DDD 阶段已产出前端项目及 `src/types.ts` 中的类型定义

你的实现必须**严格遵循** schema 中的表结构和字段命名，类型必须复用或兼容 SDD 契约。

### 本阶段任务清单
1. **搭建后端工程**：在 `backend/` 下创建独立的 Node.js + TypeScript 项目，使用 `better-sqlite3` 操作 SQLite，使用 **Vitest** 作为测试框架。
2. **先写测试，再建表**：
   - 在 `tests/db.test.ts` 中编写数据库层测试：验证建表、种子数据插入、Q 矩阵导出、X 矩阵导出
   - 运行测试，确认失败（红）
   - 实现 `src/db.ts`（建表）和 `src/seed.ts`（种子数据），使测试通过（绿）
3. **先写测试，再实现算法**：
   - 在 `tests/dina.test.ts` 中编写 DINA 模型核心测试：
     - `computeIdealResponsePattern` 的单元测试（给定 Q 和 α，验证 η 计算正确）
     - `computeLikelihood` 的单元测试（手算验证似然公式）
     - 一个极简的 2 知识点 × 4 题目 × 4 学生的确定性案例，验证算法能区分"全掌握/偏科/全不掌握"
     - 边界检查：返回的概率必须在 [0, 1]
     - 收敛性检查：算法应在限定迭代次数内完成
   - 运行测试，确认失败（红）
   - 实现 `src/algorithm/dina.ts`，使用标准 EM 算法框架：
     - E-step：基于当前 slip/guess 参数，计算每个学生属于各属性模式的后验概率
     - M-step：更新 slip/guess 参数及模式先验分布
     - 输出：每个学生在每个知识点上的边际掌握概率 P(α_k = 1)
   - 运行测试，确认通过（绿）
4. **数据库与算法整合**：
   - 在 `tests/integration.test.ts` 中编写整合测试：从真实数据库导出 Q/X 矩阵，调用 DINA 算法，验证学生画像排序（张三 > 王五）及知识点分化（李四的加减 > 乘除）
   - 实现 `src/diagnosis.ts` 作为整合层
   - 运行测试，确认通过

### 约束
- **测试先行**：每一个功能模块必须先有失败的测试，再有实现代码。Commit 历史必须能体现这一点
- 算法必须使用**数值稳定的对数似然**计算，避免连续乘法下的浮点下溢
- slip 和 guess 参数必须裁剪到合理范围 [0.01, 0.4]，防止参数退化
- 种子数据中的 X 矩阵（学生作答）必须是确定性的、可复现的，不要使用纯随机数
- 不要写 HTTP 路由、不要修改前端代码
```

### 说明
- **当时意图**：强迫 AI 严格遵循 TDD 红-绿循环，避免"先写实现再补测试"的反向流程。特别要求用确定性种子数据，确保算法测试结果可复现（后续算法调参时无需猜测"是数据变了还是代码错了"）。
- **挑战与修正**：AI 在第一次写 EM 算法时使用了直接乘法计算 likelihood，在 20 题连续相乘下出现浮点下溢（ posterior 全变 NaN）。我要求它改用对数似然 + log-sum-exp 技巧重新实现 E-step，问题得以解决。AI 还曾在 `tests/db.test.ts` 中使用下划线命名的 SQL 列名，但 TypeScript 接口是 camelCase，导致测试中断——我要求统一使用 SQL `AS` 别名映射，保持类型一致性。

### 说明
- **当时意图**：让 AI 将前三个阶段各自独立的产物（Schema、UI、算法）真正打通，验证"端到端"数据流的正确性。特别要求 LLM 建议必须有降级策略，因为在实习作业的评分环境中无法保证一定有外网 API Key。
- **挑战与修正**：AI 最初在 `server.ts` 中直接 `import type from '../frontend/src/types.ts'`，导致后端工程出现不合理的跨目录依赖。我要求它改为在后端内联定义 API 响应类型，保持后端自包含。前端 Vite 开发服务器默认端口 5173 与后端 3000 存在跨域，AI 一开始想手动处理 CORS，我引导它使用 Vite 内置的 `server.proxy` 配置，将 `/api` 透明代理到后端，这样前端代码中无需写死 `http://localhost:3000`。
