# 开发过程思路说明

## 一、从教育学业务痛点到技术任务的转化

在"AI+教育"领域，教师面临的核心痛点是**无法精准定位每个学生的知识盲区**。传统的考试只能给出总分，却无法回答"学生到底在哪些具体知识点上存在缺陷"。这种"黑盒化"的评估导致教学干预缺乏针对性——成绩差的学生被笼统地要求"多刷题"，而优势学生则无法得到有效的拔高建议。

将这一痛点转化为技术任务，关键在于引入**认知诊断（Cognitive Diagnosis）**框架：

1. **业务对象建模**：将"题目"抽象为考察一组"知识点"的测量工具（Q矩阵），将"学生作答"抽象为二元反应向量（X矩阵）。这一步直接对应 SDD 阶段的数据库 Schema 设计。
2. **算法选择**：考虑到 MVP 的轻量性要求，我选择了 DINA 模型而非更复杂的神经网络方法（如 Neural CD）。DINA 的可解释性强——每个输出概率都有明确的"属性掌握模式"语义，教师能够理解"为什么系统判断这个学生没掌握减法基础"。
3. **可视化落地**：算法产出的概率向量对非技术人员毫无意义，必须通过"认知雷达图"和"知识掌握地图"将高维诊断结果降维到人类可直观理解的视觉形态。这驱动了 DDD 阶段的 UI/UX 设计。

整个转化过程遵循"教育场景定义数据结构 → 数据结构约束算法输入 → 算法输出驱动可视化"的链路，确保技术决策始终服务于"让教师一眼看清认知盲区"这一核心目标。

## 二、与 AI 协同中遇到的典型问题

### 问题 1：AI 倾向于"一步到位"，违反阶段隔离原则

在最初的几轮对话中，AI 多次尝试在一个 Prompt 内同时完成"建表 + 画 UI + 写测试"。这会导致三个严重后果：
- Schema 未经充分推敲就被前端代码绑定，后续修改成本极高；
- UI 组件中硬编码了本应由算法生成的数据，Mock 与真实逻辑混杂；
- 测试用例沦为实现的"事后验证"，而非设计的"前置约束"。

**修正方式**：我在每个阶段的 Prompt 开头强制加入"请只聚焦于该阶段的任务，不要提前实现后续阶段的内容"这一指令，并明确要求 AI 在动手写代码前先产出设计文档（Schema 或 UI Design）。这相当于给 AI 安装了"阶段门禁"，使其输出被强制收敛到当前范式的关注点内。

### 问题 2：算法实现中的数值稳定性被忽视

在 DINA 模型的 EM 算法实现中，AI 第一版代码使用了直接乘法计算似然值：`likelihood = ∏ P(X_i | α)`。对于 20 道题目，每道题的概率约为 0.1~0.9，连续相乘后的结果会迅速下溢到 `0.0`，导致 E-step 中所有学生的后验概率变成 `NaN`，算法完全失效。

**修正方式**：我要求 AI 将对数似然（log-likelihood）引入 E-step，使用 `log-sum-exp` 技巧进行数值稳定的归一化。AI 重写后的代码在相同数据上收敛正常，且学生画像排序（张三 > 王五）符合预期。这个问题让我意识到，**算法工程师不能仅验证"逻辑正确"，还必须在浮点运算层面验证"数值正确"**——而 AI 在初次实现时往往缺乏这种底层警觉性，需要人类通过边界案例和异常输入进行引导。

### 问题 3：跨环境类型耦合导致后端无法自包含

Stage 4 整合时，AI 在 `backend/src/server.ts` 中直接写了 `import type from '../frontend/src/types.ts'`，使得后端源码依赖前端目录结构。这在 monorepo 中或许可行，但对于一个轻量级 MVP，这种跨目录引用破坏了后端的独立性和可测试性。

**修正方式**：我要求后端在 `server.ts` 中内联定义 API 响应所需的 TypeScript 接口，保持后端自包含。这一修正看似微小，实则反映了工程规范中的一个重要原则：**服务边界应当通过显式契约（如 OpenAPI Spec 或独立的类型包）来定义，而非通过文件系统的相对路径来隐式耦合**。

## 三、算法工程师如何利用 AI 提升研发效率

通过本次项目的实践，我认为算法工程师可以从以下三个维度利用 AI 提升效率：

### 1. 将 AI 作为"快速原型机"，而非"最终交付者"

AI 在生成 DINA 模型的 EM 算法框架、设计数据库 Schema、搭建前端组件等方面展现了极高的速度优势。但算法工程师的价值不在于"让 AI 写出能跑的代码"，而在于**定义问题边界、选择算法范式、验证数值正确性**。在本次项目中，AI 生成了 DINA 的 EM 骨架，但算法工程师必须决定：使用对数似然还是直接乘法？slip/guess 的裁剪阈值设为多少？最大迭代次数和收敛容差如何平衡？这些决策需要深厚的领域知识，AI 只能提供选项，不能替工程师承担责任。

### 2. 利用 AI 进行"防御性编程"的补充

AI 擅长生成"Happy Path"代码，但对边界情况、异常处理、数值稳定性往往考虑不足。算法工程师可以反向利用这一特性：先让 AI 生成核心逻辑，再刻意构造极端输入（如全对/全错的学生、空矩阵、极大/极小的概率值）来测试其脆弱点。这种"AI 写正向逻辑，工程师补逆向防护"的分工模式，能够显著提升代码的鲁棒性。

### 3. 用 AI 降低"文档与沟通"的摩擦成本

在本次项目中，AI 协助生成了 `docs/schema.md`、`docs/ui-design.md`、`prompts.md` 以及本篇文章。对于算法工程师而言，将技术方案转化为非技术人员可理解的文档往往是一项耗时且痛苦的任务。AI 可以承担"初稿生成"和"格式整理"的工作，使工程师能够将更多精力投入到模型调优和架构设计中。但同样需要强调的是，**文档中的关键决策理由、取舍分析必须由工程师亲自撰写**——因为 AI 无法真正理解业务约束和团队上下文。

## 结语

AI 不是算法工程师的替代者，而是**能力放大器**。它放大了工程师在代码生成、文档撰写、原型验证上的速度，但无法替代工程师在问题建模、算法选择、数值分析和工程决策上的核心判断力。本次 MVP 项目的最大收获，不是产出了一个可运行的认知诊断系统，而是验证了"人类定义方向、AI 填充细节、人类验收质量"这一协作模式在算法工程中的可行性。
